Learning Outcomes

This lesson closes Track 2 by examining how trained transformers produce outputs at inference time. You will master the generation loop, the KV cache mechanism, and the full spectrum of decoding strategies — then connect these to interpretability probes like surprisal analysis and output entropy tracking.

  • Write out the autoregressive factorization P(x₁,…,xₙ) = Π P(xₜ | x₁,…,xₜ₋₁) and explain why prefill and decode phases are computationally distinct.
  • Describe how the KV cache works, calculate its memory footprint for GPT-2-small at a given context length, and explain what information it stores.
  • Compare greedy decoding, temperature sampling, top-k, top-p (nucleus), and beam search on dimensions of determinism, diversity, coherence, and cost.

  • Define surprisal (−log P(token | context)) and explain how it can serve as a knowledge probe for factual completions.

  • Identify what output distribution entropy at a given step implies about the model's downstream behavior, and name two causes of sudden entropy spikes mid-generation.

Conceptual Setup

Generation mechanics describes how a trained transformer model is used at inference time to produce output sequences. For language models, this means autoregressively predicting one token at a time, each prediction conditioned on all previous tokens. Understanding generation mechanics is essential for mechanistic interpretability for three reasons:

  1. The generation loop is where model behavior manifests. All the circuit-level computation studied in Lessons 1 and 2 ultimately produces a logit distribution over the vocabulary at each step. The generation loop is the observable output of the entire model's computation.
  2. Different decoding strategies produce qualitatively different behaviors from the same model. A model that generates factually accurate text with greedy decoding may hallucinate under high-temperature sampling. Understanding decoding is understanding one degree of freedom that separates model capability from model behavior.
  3. The KV cache makes causal analysis tractable. Because the KV cache stores the K and V representations for all previous tokens, researchers can intervene at specific positions without re-running the full computation. Activation patching at generation time exploits this structure.
Track 2 Synthesis

The generation loop combines everything from Lessons 1 and 2. Attention heads (Lesson 1) route information from KV-cached past tokens to the current position. MLP layers (Lesson 2) apply factual knowledge and transformations at the current position. The final unembedding (Track 1, Lesson 4) converts the residual stream to logits. Decoding strategy determines which logit becomes the next token. This is the complete picture of autoregressive language modeling.

The KV cache is a memory-compute tradeoff: by caching the key and value matrices for all past tokens, the model avoids recomputing attention for the entire context at every generation step. Without caching, generation would require O(T²) attention computation; with caching, each new token requires only O(T) attention (one token attends to T cached keys/values). For long-context generation, this is the primary enabler of practical inference.

Intuition First

Imagine a person completing a sentence one word at a time, checking their full sentence so far before each new word. That's autoregressive generation. They don't re-read from scratch every time — they remember what they've written. That memory is the KV cache.

The KV cache is like a notepad where the model writes down what it computed about each past token (specifically: the key and value vectors for each attention head at each layer). When generating the next token, the model only needs to compute the query for the new position and look up its attention weights over the cached keys — it doesn't reprocess any previous token.

Sampling as Decision-Making

The decoding strategy is how the model "decides" which word to say next. Greedy always picks the most probable word — boring but reliable. Temperature makes the model more adventurous by flattening the distribution (high T) or sharpening it (low T). Top-p stops the model from ever considering very unlikely choices, regardless of how many tokens it would take. Beam search thinks multiple steps ahead before committing.

Intuition for top-p (nucleus) sampling: rather than always taking the top k tokens (fixed count), nucleus sampling takes the minimum set of tokens whose cumulative probability exceeds p. On a high-confidence step (e.g., "The capital of France is ___"), the top 1 token may already account for 70% of the probability, so nucleus sampling with p=0.9 would include only 3–4 tokens. On a creative or uncertain step, the top 50 tokens might collectively only account for 90%, so nucleus sampling naturally includes more options. This adaptive behavior is why top-p is preferred over top-k for most generation tasks.

Intuition for output entropy as an interpretability signal: if the model is confident (low entropy), it has strongly committed to a direction. This can be a positive signal (strong factual knowledge) or a warning sign (overconfident hallucination). If the model is uncertain (high entropy), it is in a regime where the generation will be less predictable and more sensitive to sampling stochasticity. Tracking entropy step-by-step during generation reveals when the model is "deciding" vs. "filling in" — high-entropy steps are the semantically critical decision points.

Technical Core

Autoregressive Factorization

Joint probability factorization: P(x₁, x₂, ..., xₙ) = P(x₁) · P(x₂ | x₁) · P(x₃ | x₁,x₂) · ... · P(xₙ | x₁,...,xₙ₋₁) = ∏_{t=1}^{n} P(xₜ | x₁, ..., xₜ₋₁) Each factor P(xₜ | context) is computed by: 1. Forward pass on {x₁,...,xₜ₋₁} → residual stream at position t-1 2. Unembedding: logits = residual_stream · W_U^T (V-dimensional) 3. Softmax: probs = softmax(logits) (V-dimensional distribution) 4. Sample/select: xₜ ~ probs (or argmax for greedy)

The Generation Loop with KV Cache

PREFILL PHASE: process all prompt tokens in parallel ────────────────────────────────────────────────────── Prompt: [x₁, x₂, ..., xₙ] ↓ Single forward pass (full attention, all positions) ↓ For each layer l, each head h: KV_cache[l][h] = { K: (n, d_k), V: (n, d_k) } ↓ Logits at position xₙ → first generated token x_{n+1} DECODE PHASE: one token at a time ────────────────────────────────────────────────────── For step t = n+1, n+2, ..., until EOS or max_length: Input: single token x_t (shape: (1, 1)) ↓ Embedding: e_t ∈ ℝ^d ↓ For each layer l: Q_t = e_t · W_Q (query for new position) Attend over KV_cache[l]: attn(Q_t, K_{1:t}, V_{1:t}) Update residual stream Apply MLP Append K_t, V_t to KV_cache[l] ↓ logits = residual_state · W_U^T (V = 50257 logits) ↓ Apply decoding strategy ↓ x_{t+1} ← sample / argmax ↓ (repeat)

KV Cache Memory Footprint

Memory per token (GPT-2-small): Layers: L = 12 Heads: H = 12 d_k: 64 (per head) K and V: 2 tensors dtype: float16 = 2 bytes Per-token memory = L × H × d_k × 2 × 2 bytes = 12 × 12 × 64 × 2 × 2 = 73,728 bytes ≈ 72 KB per token For 512-token context: Total KV cache = 72 KB × 512 = 36,864 KB ≈ 36 MB For GPT-3 (96 layers, 96 heads, d_k=128, float16): Per-token = 96 × 96 × 128 × 2 × 2 = 4,718,592 bytes ≈ 4.5 MB per token For 2048 tokens: ~9.2 GB KV cache ← memory bottleneck for long contexts

Output Distribution Entropy

Shannon entropy of the output distribution at step t: H_t = − Σ_{v=1}^{V} p_t(v) · log₂ p_t(v) (bits) Calibration: H = 0: perfectly certain (argmax = 1.0) H = 1 bit: two tokens each at ~50% H = log₂(V): fully uniform over V tokens (V = 50257 → 15.6 bits for GPT-2) Practical ranges in GPT-2-small: Factual completions: H ≈ 1–4 bits (confident) Common continuations: H ≈ 4–8 bits (moderate) Open-ended positions: H ≈ 8–12 bits (uncertain) Degenerate state: H > 12 bits (near-uniform, possible OOD)
Repetition Penalty

A common generation artifact is repetitive output (the model keeps generating the same phrase). This happens because once a token is generated, its key-value entries in the KV cache provide a strong attractor for the same token to appear again. Repetition penalty (Keskar et al. 2019) addresses this by scaling down logits for tokens that have already appeared: logit(v) /= penalty if v in past tokens (penalty > 1 reduces probability of repeats).

Worked Example

We implement a manual generation loop with per-step log-probability and entropy logging, then compare greedy and nucleus sampling on a factual prompt.

  1. 01 Implement a manual generation loop that does NOT call model.generate() — instead, call model(generated) at each step, extract logits, and choose the next token manually.
  2. 02 At each step, record: (a) the greedy-selected token and its log-probability; (b) the entropy of the full distribution; (c) the top-5 candidate tokens and their probabilities.
  3. 03 Compare the entropy profile for a factual prompt ("The Eiffel Tower is located in") vs. a creative prompt ("Once upon a time") — factual should show lower entropy at the key decision step. 
import torch
from transformers import GPT2LMHeadModel, GPT2Tokenizer
import math
model = GPT2LMHeadModel.from_pretrained("gpt2")
tok = GPT2Tokenizer.from_pretrained("gpt2")
model.eval()
def entropy_bits(probs):
    """Shannon entropy in bits."""
    eps = 1e-10
    return -torch.sum(probs * torch.log2(probs + eps)).item()
def manual_generate(prompt, n_steps=10, strategy="greedy", temperature=1.0, top_p=0.9):
    input_ids = tok.encode(prompt, return_tensors="pt")
    generated = input_ids.clone()
    history = []
    for step in range(n_steps):
        with torch.no_grad():
            out = model(generated)
        logits = out.logits[0, -1, :]       # logits for next token
        # Temperature scaling
        if temperature != 1.0:
            logits = logits / temperature
        probs = torch.softmax(logits, dim=-1)
        H = entropy_bits(probs)
        top5 = probs.topk(5)
        top5_tokens = [tok.decode([i]) for i in top5.indices]
        if strategy == "greedy":
            next_token = probs.argmax().unsqueeze(0)
        elif strategy == "top_p":
            # Nucleus sampling: keep tokens until cumulative prob > top_p
            sorted_probs, sorted_idx = probs.sort(descending=True)
            cumulative = sorted_probs.cumsum(dim=-1)
            mask = cumulative - sorted_probs > top_p
            sorted_probs[mask] = 0.0
            sorted_probs /= sorted_probs.sum()
            next_sorted = torch.multinomial(sorted_probs, 1)
            next_token = sorted_idx[next_sorted]
        else:
            next_token = torch.multinomial(probs, 1)
        log_prob = math.log(probs[next_token[0]].item() + 1e-10)
        token_str = tok.decode([next_token[0].item()])
        history.append({
            'token': token_str,
            'log_prob': log_prob,
            'entropy': H,
            'top5': list(zip(top5_tokens, top5.values.tolist()))
        })
        generated = torch.cat([generated, next_token.unsqueeze(0)], dim=1)
    return history
# Run on factual prompt
factual = "The Eiffel Tower is located in"
hist_factual = manual_generate(factual, n_steps=8, strategy="greedy")
print("Factual prompt — greedy generation:")
print(f"{'Step':>4}  {'Token':>12}  {'log_p':>7}  {'H(bits)':>8}  Top-2")
for i, h in enumerate(hist_factual):
    top2 = [f"{t!r}={v:.3f}" for t, v in h['top5'][:2]]
    print(f"{i+1:>4}  {h['token']!r:>12}  {h['log_prob']:>7.3f}  {h['entropy']:>8.2f}  {', '.join(top2)}")
# Run on creative prompt for comparison
creative = "Once upon a time"
hist_creative = manual_generate(creative, n_steps=8, strategy="greedy")
print("\nCreative prompt — greedy generation:")
print(f"{'Step':>4}  {'Token':>12}  {'log_p':>7}  {'H(bits)':>8}")
for i, h in enumerate(hist_creative):
    print(f"{i+1:>4}  {h['token']!r:>12}  {h['log_prob']:>7.3f}  {h['entropy']:>8.2f}")
# Step 2: Nucleus sampling comparison
# High-level API for convenience
output = model.generate(
    tok.encode(factual, return_tensors="pt"),
    max_new_tokens=20,
    do_sample=True,
    top_p=0.9,
    temperature=0.8,
    repetition_penalty=1.1,
    pad_token_id=tok.eos_token_id
)
print("\nTop-p (0.9) + temp (0.8) generation:")
print(tok.decode(output[0], skip_special_tokens=True))
# Expected: factual prompt step 1 should have very low entropy (< 4 bits)
# and log_prob close to 0 for "Paris" or "the city of Paris"
# Creative prompt should show H > 8 bits at most steps

Typical observation: for "The Eiffel Tower is located in", the greedy first token is " Paris" or " the" with log_prob ≈ −0.5 to −1.2 and entropy ≈ 2–5 bits. For "Once upon a time", first generated tokens like " there", " a", or " in" show entropy ≈ 8–11 bits — the model is much less certain about what comes next.

Mechanistic Interpretation Layer

Text: Surprisal as a Knowledge Probe

What is measured: The surprisal −log₂ P(token | context) for the final token in a factual completion. For "The capital of France is Paris" vs. "The capital of France is Berlin", the correct completion should have lower surprisal (higher probability). The difference in surprisal is the model's "knowledge signal."

The LAMA benchmark: Petroni et al. 2019 (LAMA: Language Model Analysis) systematically tested whether LLMs have factual knowledge by measuring the log-probability of correct vs. incorrect factual completions. GPT-2-large assigns log P(" Paris") ≈ −0.84 given "The capital of France is", vs. log P(" Berlin") ≈ −3.2 — a clear knowledge signal of ~2.4 log-units.

Why it matters: Surprisal provides a continuous, differentiable measure of model confidence in factual claims. It can be computed without any model modification or intervention. The surprisal profile across a full sentence (per-token surprisal) reveals which tokens are unpredictable and therefore informationally dense.

Caution: High confidence does not equal truth. Models can be confidently wrong (hallucination). Log-probability is also influenced by token frequency in training data — a rare but correct completion may have higher surprisal than a common but incorrect one. Additionally, surprisal is a function of the tokenizer: multi-token words ("photosynthesis" → ["photo", "synth", "esis"]) have lower per-token surprisal than single-token words because the subsequent tokens are highly predictable given the first. Always normalize for tokenization artifacts when comparing surprisal across different words.

Vision: Autoregressive Image Generation Entropy

What is measured: Per-step entropy of the pixel distribution in autoregressive image models (iGPT / ImageGPT). These models generate images in raster-scan order (left to right, top to bottom), treating pixel values as a discrete vocabulary (quantized to 512 colors). The entropy of the distribution at each pixel position reflects the model's uncertainty.

Entropy as a visual "difficulty map":

  • Smooth regions: Very low entropy — the next pixel is highly predictable given neighboring pixels of the same color/texture. The model has essentially memorized the texture pattern.
  • Edges and boundaries: Higher entropy — the model is making a harder decision about whether the region transitions to a new object or continues the current one.
  • Object interiors at early steps: Very high entropy — before any context from the object has been generated, the model is highly uncertain.

Why it matters: The entropy map serves as an implicit saliency map — regions of high entropy are where the model is making the hardest decisions. This connects visual uncertainty to attention head behavior (which patches contribute information to these high-entropy decisions?).

Caution: Pixel-level entropy in autoregressive image models reflects low-level statistical structure (texture regularity, color continuity) rather than semantic content. A uniform-colored wall has very low entropy not because it is semantically unimportant, but because its texture is trivially predictable. Do not equate low generation entropy with semantic unimportance.

Speech: TTS Autoregressive Generation (VALL-E)

What is measured: Per-step token probability and attention patterns in VALL-E (Wang et al. 2023), a neural TTS model that generates speech by autoregressively producing acoustic codec tokens conditioned on text and a speaker prompt.

How VALL-E works mechanistically:

  • Input: Text tokens + speaker prompt audio (encoded as codec tokens).
  • Generation: Autoregressively produce acoustic codec tokens. Each token represents ~12.5ms of audio at multiple quantization levels (RVQ codes).
  • Attention function: Cross-attention heads copy speaker-specific style features from the prompt (timbre, prosody, speaking rate). Self-attention heads implement text-to-speech alignment — each decoder step attends to the relevant portion of the text.

Mechanistic question: Which attention heads in VALL-E implement text-to-speech alignment vs. speaker style copying? Preliminary analysis suggests that different decoder heads specialize in these two functions, analogous to the specialization of attention heads in language models.

Caution: TTS models are multimodal input-output — the input includes both text (discrete tokens) and audio (codec tokens), and the output is audio. Causal tracing in this setting is more complex because interventions on text tokens affect a different modality than interventions on audio prompts. Standard activation patching from LM research requires adaptation for multimodal settings.

Hands-on Exercise

Exercise: Compare Decoding Strategies on Diverse Prompts

Task: Compare greedy, temperature sampling, and top-p sampling on 3 different prompts and analyze the entropy and log-probability profiles.

  1. 01 Choose 3 prompts: (a) Factual — "The speed of light is approximately"; (b) Creative — "Once upon a time in a land far"; (c) Technical — "import numpy as np\n# This function computes"
  2. 02 Generate with 4 strategies: Greedy (T=1, deterministic), Low temperature (T=0.3), High temperature (T=1.2), Top-p (p=0.9, T=1.0). For each strategy and prompt, generate 15 tokens.
  3. 03 Record per-step metrics: Log-probability of the selected token, entropy of the full distribution (before truncation/sampling), and the text of the selected token.
  4. 04 Compute step-1 entropy: For each prompt, measure the entropy of the distribution at step 1 (before any generation). This is the model's "prior uncertainty" for each prompt type. Rank the 3 prompts by step-1 entropy.
  5. 05 Coherence assessment: Read the 4 outputs for each prompt. Rate each output's factual accuracy / coherence on a 1–3 scale. Does greedy always win on factual prompts? Does top-p win on creative prompts?
  6. 06 KV cache analysis: For the factual prompt, run the generation twice — once with the standard generation loop and once by manually recomputing from scratch at each step (no KV cache). Verify the outputs are identical. Time both approaches: the manual recomputation should be significantly slower for long sequences.

Expected output: Factual prompt step-1 entropy should be substantially lower than creative prompt (4 vs. 10+ bits). Greedy should achieve highest log-probability per step but lowest diversity. T=1.2 will produce the most creative but occasionally incoherent outputs.

Success criteria: Correctly identify the highest-entropy prompt at step 1. Demonstrate that greedy and T=0.3 produce identical or near-identical outputs on the factual prompt (both converge to the high-probability token). Verify that top-p with p=0.9 produces more diverse outputs than greedy across 5 repeated runs.

Reflection: Why might top-p be preferred over top-k for most tasks? When would beam search be better than sampling? Under what conditions would you use a repetition penalty?

Stretch: Implement a minimal contrastive decoding variant: run the factual prompt through both GPT-2-small and GPT-2-large. Subtract the small model's log-probs from the large model's log-probs before sampling. Observe if factual accuracy of "The speed of light is ___" improves. (Correct: ~299,792 km/s or 3×10⁸ m/s)

Common Confusion

Temperature scaling and top-p are applied sequentially, not as alternatives. A typical generation setup uses temperature first (rescale logits) and then top-p (truncate the tail). Applying both simultaneously is valid and common. Greedy decoding is equivalent to temperature → 0 (argmax) with top-p = 1.0 (no truncation).

Real Model Artifact

Quantitative benchmarks from published work on GPT-2 and nucleus sampling:

GPT-2 Generation Benchmarks

GPT-2-small on Penn Treebank: mean per-token log-probability ≈ −3.56 (perplexity ≈ 35). On WebText held-out data: perplexity ≈ 29.4. These are lower bounds on surprisal for in-distribution text — out-of-distribution prompts (technical documentation, mathematical notation) will show substantially higher surprisal.

Factual Completion Log-Probabilities

For GPT-2-large on LAMA factual probes: log P(" Paris") ≈ −0.84 given "The capital of France is". This corresponds to P ≈ 0.43 — the model assigns 43% probability to " Paris" (correct). The runner-up candidates (" Lyon" ≈ 0.12, " France" ≈ 0.08) have substantially lower probability, indicating a clear knowledge signal. For less well-represented facts (e.g., capital of Burkina Faso), the correct answer may not even appear in the top-10 predictions.

KV cache memory calculation for GPT-2-small at 512 tokens:

  • 12 layers × 12 heads × 64 d_k × 2 (K and V) × 2 bytes (float16) × 512 tokens
  • = 12 × 12 × 64 × 2 × 2 × 512 = 18,874,368 bytes ≈ 18.9 MB

Holtzman et al. 2020 "The Curious Case of Neural Text Degeneration" introduced nucleus sampling (top-p) and demonstrated quantitatively that greedy and beam search degenerate on long-form generation — they produce repetitive text that diverges from human writing quality after ~100 tokens. Nucleus sampling maintains the "human-like" distribution because it does not always select the mode, allowing controlled exploration of the distribution tail.

Speculative Decoding (Chen et al. 2023): A generation efficiency technique where a small "draft" model generates k tokens speculatively, and the large "verifier" model checks all k tokens in parallel (exploiting the prefill phase's parallelism). If the verifier agrees, all k tokens are accepted in one step — effectively k× throughput improvement at matched quality. This works because the KV cache can be computed in parallel for the draft tokens.

Assessment Check

Scenario 1: Confident Wrong vs. Confident Right

A model generates "The Eiffel Tower is in Paris" with log P ≈ −0.5 (very high confidence) but generates "The Eiffel Tower was built in 1887" with log P ≈ −1.2, when the historically correct year is 1889. What does this suggest about where and how the model stores different types of knowledge?

Show strong answer

Strong answer: The high-confidence Paris completion (log P ≈ −0.5) suggests that the "Eiffel Tower → Paris" association is strongly encoded in the model's MLP memories, and the attention routing from "Eiffel Tower" to "Paris" has been reinforced by many training examples. This is an often-repeated, prominent fact in web text.

The incorrect year (1887 vs 1889) with lower confidence (log P ≈ −1.2) suggests several possible failure modes:

  1. Training frequency: The exact construction year may appear inconsistently in training data (some sources say "began construction in 1887" while others say "completed in 1889"). The model has learned a mixture of these, with the more common surface form winning. Competing log-probabilities: log P("1889") vs. log P("1887") may be closer than log P("Paris") vs. log P("Berlin").
  2. Interference from nearby dates: The model may have confounded 1887 (start of construction) with 1889 (completion / Exposition Universelle). These co-occur in many texts about the Eiffel Tower, creating ambiguity in the factual neuron encoding.
  3. Test with logit lens: Run the logit lens on the year prompt. Does " 1887" vs. " 1889" already diverge at early layers, or do they converge to " 1887" only in final layers? Early divergence suggests the wrong fact is deeply encoded; late divergence suggests the correct fact might be recoverable with better prompting.

Key takeaway: confidence (log-probability) reflects training frequency as much as factual accuracy. High confidence is not a reliable truthfulness signal.

Scenario 2: Entropy Spike at Step 50

During inference, a model's output distribution at step t=50 suddenly has entropy H > 12 bits (near-uniform over the 50,257-token vocabulary), even though all previous steps had H ≈ 4–8 bits. The sequence is coherent up to step 49. What might cause this, and how would you diagnose it?

Show strong answer

Strong answer: A sudden entropy spike at a specific step after a period of coherent low-entropy generation is anomalous and warrants diagnostic investigation. Possible causes:

  1. Out-of-distribution generated token: A token generated at step 48 or 49 may be outside the model's training distribution (e.g., a rare token or an unusual token combination), corrupting the residual stream state and leading to incoherent processing at step 50. Check: does removing the step 49 token and regenerating restore low entropy?
  2. KV cache truncation: If the context window is exceeded and the KV cache is truncated (oldest tokens removed), the model may lose access to the document context that was providing the discourse coherence signal. Check: what is the total sequence length at step 50? Is it near the model's context window limit (1024 for GPT-2)?
  3. Repetition penalty side effect: If a strong repetition penalty is applied, and by step 50 all common continuations have been penalized, the remaining token distribution may be artificially flattened. Check: generate without repetition penalty and compare entropy profiles.
  4. Discourse topic exhaustion: The model may have genuinely run out of topic — it has said everything it knows about the subject and is now uncertain about what to generate next. This is a semantically valid high-entropy state, not a computational failure.

Diagnosis steps: (1) Run logit lens at step 50 — is entropy high at all layers (suggesting deep uncertainty) or only at the final layer (suggesting late-layer confusion)? (2) Inspect attention patterns at step 50 — is attention degenerate (attention sink for all heads)? (3) Check if the issue is reproducible with temperature=0 (greedy) — if entropy is still high, it is not a sampling artifact.

Scenario 3: Beam Search vs. Sampling for Factual Tasks

A researcher proposes using beam search (beam width = 10) instead of greedy decoding for factual question answering, arguing that beam search "looks ahead" and will find the globally highest-probability completion rather than the locally optimal greedy choice. Is this reasoning correct? Under what conditions would beam search actually help?

Show strong answer

Strong answer: The reasoning contains a valid insight but the practical benefit depends on the specific task:

  1. When beam search helps: Beam search benefits tasks where the locally optimal first token is not the globally optimal first token. This happens when the answer requires a specific phrasing that starts with a low-probability token. Example: "What year was the Eiffel Tower completed?" — if the answer is "It was completed in 1889," the word "It" may have lower local probability than "The" or "In", but the full sequence "It was completed in 1889" may have higher joint probability. Beam search can find this.
  2. When beam search does NOT help for factual QA: For short factual completions (1–3 tokens), the greedy choice is usually optimal or near-optimal. The factual token has high local probability precisely because the model has learned to associate the prompt pattern with that token. Beam search with width 10 adds compute but minimal accuracy gain for "The capital of France is ___" — greedy is already finding " Paris."
  3. Beam search failure mode: For open-ended generation, beam search tends to produce repetitive text (it finds the "highest joint probability" sequence, which is often a repeated high-probability phrase). Holtzman et al. 2020 demonstrated this experimentally — beam search outputs are often degenerate on long-form tasks.
  4. Practical recommendation: Use greedy for very short factual completions (<5 tokens). Use beam search for structured outputs where global coherence matters (translation, structured extraction). Use top-p for longer free-form generation.
