Learning Outcomes

You will leave able to implement clean/corrupt forward passes, write patching hooks, compute recovery scores, and interpret the resulting causal heatmaps.

  • Define the recovery score and explain why values near 0 and 1 have different interpretations.

  • Distinguish between noising and denoising patching and explain when each is appropriate.

  • Design a well-controlled patching experiment with a clean corruption strategy.

Conceptual Setup

Activation patching (also called causal tracing or interchange intervention) is the gold standard for establishing causal claims in mechanistic interpretability. While probing tells us what information is encoded, activation patching tells us what information is actually used in computation.

The core technique: Run two forward passes — one on a "clean" input (where the model gives the correct answer) and one on a "corrupted" input (where the model gives a wrong answer). Then patch specific activations from the clean run into the corrupted run and measure whether the output recovers toward the clean answer. Recovery implies causal necessity of the patched component.

This technique is grounded in the causal inference framework. "Patching" is equivalent to a "do-intervention": we are forcing a specific component to take a particular value and observing the downstream effect. Unlike correlation-based approaches (probing), this directly tests causal relationships.

The key design choices are: (1) what to use as the "clean" input, (2) how to corrupt the input — corruption should specifically disrupt the feature of interest without changing unrelated features, (3) which components to patch — you can patch at any granularity from the full residual stream to individual attention head outputs, and (4) how to measure recovery — typically as a normalised difference between clean and corrupt logit scores.

Intuition First

Imagine two factory lines producing the same product, but one line produces a defective item. You want to find which machine on the second line caused the defect.

  1. Identify the good line (clean run): The first factory line produces a perfect item. Every machine output on this line is correct.
  2. Identify the defective line (corrupted run): The second factory line has a defect somewhere. The final product is wrong.
  3. Swap one component at a time: Take machine A's output from the good line and install it in the same position on the defective line. Does the product improve?
  4. Find the causal machine: If swapping machine A fully fixes the defect, machine A is causally responsible. If it does nothing, machine A is not the culprit.
The analogy: "Clean" = correct input/output pair. "Corrupted" = input designed to cause wrong output. "Swapping machine A" = patching one layer's activations from clean to corrupted. "Does the product improve?" = does the output logit for the correct answer recover?

Crucially, you are not training anything or approximating anything. You are directly intervening on a running computation and observing the effect. This is the cleanest possible causal test available without access to training data or model weights.

Technical Core

Setup and Recovery Score

Let x_clean be the clean input and x_corrupt be the corrupted input. For a target token t* (the correct answer):

# Recovery score for component c:
effect(c) = (logit_patched_c(t*) - logit_corrupt(t*)) /
            (logit_clean(t*) - logit_corrupt(t*))

# Interpretation:
# effect = 0.0  -> patching this component has no effect (not causal)
# effect = 0.5  -> patching partially restores correct prediction
# effect = 1.0  -> patching fully restores correct prediction (causally sufficient)
# effect > 1.0  -> patching overcorrects (component was suppressing the answer)

What to Patch

  • Full residual stream at layer l: Measures total causal contribution of all computations up to layer l at a given token position.
  • Attention head output (head h, layer l): Measures whether this specific attention head is causally responsible.
  • MLP output at layer l: Measures whether the MLP at this layer stores or processes the relevant information.
  • Attention pattern (Q, K, V): Measures whether specific attention routing is causal.

Noising vs. Denoising

DirectionMethodQuestion AskedUse Case
DenoisingPatch FROM clean INTO corrupted"Is this component sufficient to restore correct behavior?"Standard causal tracing (ROME)
NoisingPatch noise INTO clean run"Is this component necessary for correct behavior?"Finding critical components

Path Patching

Standard activation patching patches at a single point in the computation graph. Path patching (Goldowsky-Dill et al. 2023) restricts patches to specific information paths — for example, patching only the signal flowing from attention head A's output to attention head B's key input, while leaving all other paths unchanged. This allows finer-grained decomposition of information flow.

Comparison of Intervention Methods

MethodWhat Is TestedPrecisionComputational Cost

Worked Example

A complete layer-by-layer activation patching experiment on GPT-2, testing which layers are causally responsible for retrieving factual associations.

import torch
import numpy as np
from transformers import GPT2LMHeadModel, GPT2Tokenizer

model = GPT2LMHeadModel.from_pretrained("gpt2")
tok = GPT2Tokenizer.from_pretrained("gpt2")
model.eval()

clean_text = "The Eiffel Tower is in"
corrupt_text = "The Golden Gate Bridge is in"  # different subject -> different answer

clean_ids = tok(clean_text, return_tensors="pt")
corrupt_ids = tok(corrupt_text, return_tensors="pt")

target_token = tok.encode(" Paris")[0]

# Step 1: Save clean activations via forward hooks
clean_hidden = {}
def save_hook(layer_idx):
    def hook(module, inp, out):
        clean_hidden[layer_idx] = out[0].detach()
    return hook

hooks = []
for l in range(12):
    h = model.transformer.h[l].register_forward_hook(save_hook(l))
    hooks.append(h)

with torch.no_grad():
    clean_out = model(**clean_ids)
clean_paris_logit = clean_out.logits[0, -1, target_token].item()
for h in hooks:
    h.remove()

# Step 2: Get corrupted baseline
with torch.no_grad():
    corrupt_out = model(**corrupt_ids)
corrupt_paris_logit = corrupt_out.logits[0, -1, target_token].item()

print(f"Clean logit for Paris:   {clean_paris_logit:.3f}")
print(f"Corrupt logit for Paris: {corrupt_paris_logit:.3f}")

# Step 3: Patch each layer one at a time
print("\\nRecovery scores by layer:")
for patch_layer in range(12):
    def patch_hook(module, inp, out):
        patched = list(out)
        # Patch last token position from clean run
        patched[0] = patched[0].clone()
        patched[0][0, -1, :] = clean_hidden[patch_layer][0, -1, :]
        return tuple(patched)

    h = model.transformer.h[patch_layer].register_forward_hook(patch_hook)
    with torch.no_grad():
        patched_out = model(**corrupt_ids)
    h.remove()

    patched_logit = patched_out.logits[0, -1, target_token].item()
    recovery = (patched_logit - corrupt_paris_logit) / (clean_paris_logit - corrupt_paris_logit)
    bar = "#" * int(max(0, recovery) * 20)
    print(f"  Layer {patch_layer:2d}: recovery={recovery:.3f}  {bar}")
Expected output: Recovery scores should be low for early layers (0–4), rise in middle layers (5–9), and peak at one or two layers. The peak identifies the layer most causally responsible for storing the Paris association.

Mechanistic Interpretation Across Modalities

Text: The ROME Experiment (Meng et al. 2022)

For the prompt "The Eiffel Tower is in ___", activation patching on GPT-2-XL was performed across all layers and all token positions. The result was a heatmap showing recovery score at each (layer, token position) coordinate.

Key finding: Layers 15–17, specifically at the last subject token position ("Tower"), were most causally important. Patching these activations fully restored the "Paris" prediction even when the subject was corrupted with Gaussian noise.

Why it matters: This was the first definitive localisation of factual memory in a large language model — identifying not just the layer but the precise token position responsible for factual retrieval.

Caution: This result is specific to GPT-2-XL and to this type of subject-relation-object factual query. Different model architectures, different fact types, and different relation types may show different localisation patterns. Do not over-generalise.

Vision: Activation Patching in ViT for Object Recognition

Patch activations from an image containing a dog into an image without a dog, at different ViT layers. Measure recovery of the "dog" class logit.

  • Early layers (1–4): Low recovery — low-level features are shared across many classes.
  • Middle layers (5–8): Recovery rises as object parts become discriminative.
  • Late layers (8–10): High recovery — object identity strongly localized here.

What is measured: Recovery score for "dog" class logit per layer per patch token position.

Caution: ViT processes patches in parallel, so "token position" in the patching experiment must correspond to a meaningful image patch location (e.g., the patch containing the dog). Averaging over all patch positions gives a diluted signal.

Speech: Activation Patching for Voicing Distinction

Run two utterances — one with /p/ (unvoiced) and one with /b/ (voiced, identical articulation otherwise). Patch residual stream from the /b/ utterance into the /p/ computation at each layer.

Expected finding: Recovery of "voiced" classification is causal in layers 3–6 of wav2vec2. This is consistent with the probing result (voicing is linearly decodable in those layers) and confirms that the encoding is causally used.

Caution: Speech signals are continuous and time-aligned. The "token position" for speech patching must correspond to the correct audio frame (the voicing onset frame). Misalignment by even one frame can wash out the effect.

Hands-on Exercise

Simple Activation Patching on GPT-2

Task: Identify which layers are causally responsible for a factual prediction in GPT-2.

  1. Choose clean input: "The president of the United States is" (target: " Biden" or another factual answer).
  2. Choose corrupted input: "The CEO of Google is" (different entity, different answer).
  3. Run clean forward pass. Save all 12 layer outputs using forward hooks.
  4. Run corrupted forward pass. Record the target token logit.
  5. For each layer, patch the clean hidden state into the corrupted run at the last token position only.
  6. Compute recovery score for each layer: (patched logit − corrupt logit) / (clean logit − corrupt logit).
  7. Plot recovery score vs. layer index. Identify the peak recovery layer.

Expected output:

  • Recovery should peak at middle-to-late layers (8–11 for this type of factual query in GPT-2-small).
  • Early layers (0–4) should show near-zero recovery.
  • The peak recovery layer is where factual information is "retrieved" into the prediction stream.

Reflection questions:

  • Why does patching early layers produce less recovery than patching late layers? What does this tell you about when factual information is "retrieved"?
  • If you patch at layer 5 and get recovery 0.3, and at layer 9 you get recovery 0.8, what does this suggest about the layerwise computation?
Stretch challenge: Do position-resolved patching. Instead of patching at the last token position, patch only at the "United States" token position. Does recovery change? Now patch only at the "president" position. Which position gives higher recovery — the relation-describing token or the subject-final token? This replicates the key insight of Meng et al. 2022.

Real Model Artifact

Meng et al. (2022) — ROME: Locating and Editing Factual Associations in GPT
Dataset: 1,209 subject-relation-object triples from CounterFact dataset
Model: GPT-2-XL (1.5B parameters, 48 layers)

Key quantitative findings from the activation patching analysis:

  • 78% of factual associations are most strongly localised in a contiguous band of middle MLP layers (layers 15–17 for GPT-2-XL).
  • 89% of the causal effect comes from the "last subject token" position — the final token of the subject entity span.
  • Attention layers show a secondary effect in early-middle layers (5–10), interpreted as the model "routing" the subject representation to the prediction position.
  • The combined (layer 15–17 MLP) + (early attention) circuit accounts for >95% of the total causal effect.

These findings formed the empirical basis for the ROME model editing algorithm: to change a factual association, surgically modify the MLP weight matrices in layers 15–17 at the last-subject-token position. ROME achieves targeted edits with minimal effect on other facts.

Assessment Check

Scenario 1: Interpreting Recovery Scores

You find recovery = 0.4 at layer 6 MLP, recovery = 0.7 at layer 9 MLP, and recovery = 0.2 at layer 9 attention. What can you conclude about the causal structure?

Show strong answer

Layer 9 MLP is the primary causal location. Recovery of 0.7 means patching this component alone restores 70% of the gap between corrupt and clean outputs. Layer 6 MLP contributes (~40%) but is not alone sufficient. Layer 9 attention contributes minimally (20%) — it may route information but does not store the fact.

Combined patching prediction: Patching layers 6 MLP + layer 9 MLP simultaneously would likely give recovery > 0.7 (possibly near 0.9+) since their contributions appear partially additive.

What this tells you: Factual storage is partially distributed, with a dominant location at layer 9 MLP. This is consistent with the ROME finding that factual associations are concentrated in middle-layer MLPs but with some spread across nearby layers.

Scenario 2: Recovery Near 1.0 Everywhere

You try to use activation patching to find the circuit for generating grammatically correct plurals. You find recovery near 1.0 at every layer. What does this mean and how would you redesign the experiment?

Show strong answer

Recovery near 1.0 everywhere means the information is either redundantly encoded across all layers, or the corruption does not effectively disrupt the plural feature. If every layer can restore plural generation, it means many layers contain sufficient information — which is not informative about which layer is primarily responsible.

Likely cause: The task (plurality) may be too easy — the model can re-derive it from many cues (the noun form, surrounding context, etc.) even after corruption at any single layer.

Redesign: Use a more targeted corruption strategy. Instead of corrupting the entire input, specifically disrupt only the number-encoding dimensions of the hidden state, using a known linear direction for grammatical number (found via a probe). Patch only those dimensions. This "surgical corruption" will reveal which layer specifically routes the number feature, rather than giving every layer an opportunity to re-derive it from context.
