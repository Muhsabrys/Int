Circuits

Track 3: Interpretability
4 of 8
Attention Mechanics, Activation Patching
75 min
Text · Vision · Speech

Learning Outcomes

  • Define a "circuit" in a transformer and explain what minimal means in this context
  • Describe the Indirect Object Identification (IOI) circuit and name its component heads
  • Explain the difference between correlation-based and causal circuit finding methods
  • Apply path patching to test whether a hypothesised circuit is sufficient and necessary
  • Identify circuit motifs: induction heads, name-mover heads, inhibition heads
  • Explain why circuits can fail to generalise across prompt templates
  • Discuss evidence for and against circuits in ViT and speech models

Conceptual Setup

A circuit in mechanistic interpretability is a minimal subgraph of a neural network's computational graph that is sufficient to explain a specific model behaviour. The term was introduced systematically in Elhage et al. 2021 and operationalised in Wang et al. 2022's analysis of the Indirect Object Identification (IOI) task in GPT-2.

The central question circuit analysis asks is: which components — attention heads, MLP layers, residual stream directions — are causally necessary and jointly sufficient for the model to produce a given output? This is stronger than asking which components correlate with the output. Correlation can arise from many sources: shared upstream inputs, indirect effects, or chance co-activation. Causal necessity requires that removing the component degrades the behaviour; sufficiency requires that the component set alone can reproduce the behaviour.

Circuit analysis is the highest-resolution interpretability technique currently available for transformers. It can produce mechanistic explanations at the level of individual attention heads and specific residual stream directions. Its limitation is labour intensity: identifying a circuit for a non-trivial behaviour can take weeks of iterative experimentation, and circuits discovered on one prompt template may not generalise to superficially similar templates.

Why circuits matter

If we can identify circuits, we can predict model behaviour on novel inputs, surgically edit model behaviour by intervening on circuit components, and check whether safety-relevant behaviours are implemented robustly or via brittle shortcuts.

Intuition First

Imagine a large factory with hundreds of machines. When a defect appears in the final product, you want to find the minimal set of machines responsible. You don't want to shut down the whole factory — you want to trace the defect back to its source. Circuit analysis is exactly this: tracing a model behaviour back to its mechanistic source.

The key insight is that transformers are linear in their residual stream. Every attention head and every MLP layer adds a vector to the residual stream. The final logits are a linear function of this stream. This means we can ask: "what did attention head 9.6 contribute to the final prediction of 'Paris'?" and get a concrete, interpretable answer: it added a vector in the direction of the "Paris" token in embedding space.

Circuits are identified by: first finding which components have high direct logit attribution (correlation step), then confirming with activation patching that each component is causally necessary, and finally testing that the set together is sufficient by running a "circuit-only" forward pass where all other components are ablated.

Technical Core

Direct Logit Attribution (DLA) decomposes the logit of a target token into contributions from each component:

logit(token) = W_U · x_L
             = W_U · (x_0 + Σ_l Δ_attn_l + Σ_l Δ_mlp_l)
             = W_U·x_0 + Σ_l (W_U·Δ_attn_l) + Σ_l (W_U·Δ_mlp_l)
                          ↑ each term is a scalar contribution

Path Patching tests causal importance of a specific edge (h→node) in the computation graph. Run clean and corrupted forward passes. Patch the activation of component h from clean into corrupted, and measure how much the output recovers. Full recovery → h is sufficient. No recovery → h is not on the relevant path.

Clean run:   x_clean → [model] → logit_clean
Corrupted:   x_corr  → [model] → logit_corr   (low score)
Patched:     x_corr  → [model with h patched from clean] → logit_patch

Causal effect of h = (logit_patch - logit_corr) / (logit_clean - logit_corr)

The IOI Circuit (Wang et al. 2022) — for sentences like "When Mary and John went to the store, John gave a bag to ___", GPT-2 predicts "Mary" (the indirect object). The circuit involves 26 attention heads grouped into 5 functional classes:

Head ClassCountFunctionLayers
Name-mover heads3Copy "Mary" token to final position9, 10
Backup name-mover3Redundant copy of indirect object10, 11
Negative name-mover2Suppress "John" (S-inhibition)10, 11
Induction heads6Detect repeated name pattern5, 6
Duplicate token heads3Flag positions where S appears twice0, 1, 2
Sufficiency vs Necessity

The IOI circuit is necessary: ablating all 26 heads drops accuracy from 92% to near chance. It is sufficient: running only those 26 heads (ablating all others) maintains 86% accuracy. The gap between 86% and 92% is explained by backup heads and distributed processing.

IOI Circuit Data Flow:

[Subject token S appears twice]
        ↓
Duplicate-token heads (L0-2): flag S as duplicated
        ↓
Induction heads (L5-6): detect AB...A pattern
        ↓
S-inhibition heads (L7-8): suppress output of S
        ↓
Name-mover heads (L9-10): copy IO (Mary) to final position
        ↓
Logit of "Mary" boosted → model predicts "Mary"

Worked Example

Computing Direct Logit Attribution for the IOI task:

import torch
from transformers import GPT2LMHeadModel, GPT2Tokenizer

model = GPT2LMHeadModel.from_pretrained("gpt2")
tok   = GPT2Tokenizer.from_pretrained("gpt2")
model.eval()

text = "When Mary and John went to the store, John gave a bag to"
inputs = tok(text, return_tensors="pt")
io_token = tok.encode(" Mary")[0]   # the target token

# Full forward pass with all intermediate outputs
with torch.no_grad():
    out = model(**inputs, output_hidden_states=True, output_attentions=True)

W_U = model.lm_head.weight  # (50257, 768)
hidden_states = out.hidden_states  # 13 tensors of shape (1, T, 768)

# Compute per-layer contribution to logit of "Mary"
contributions = []
prev = hidden_states[0][0, -1, :]  # embedding layer
for l in range(1, 13):
    curr = hidden_states[l][0, -1, :]
    delta = curr - prev
    contrib = (W_U[io_token] @ delta).item()
    contributions.append((f"Layer {l}", contrib))
    prev = curr

contributions.sort(key=lambda x: x[1], reverse=True)
for name, val in contributions[:5]:
    print(f"{name}: +{val:.3f}")
# Expected: layers 9, 10 dominate (name-mover heads)

Mechanistic Interpretation Layer

Text Models

The IOI circuit is the canonical example of circuit analysis in text models. Beyond IOI, identified circuits include: the greater-than circuit (Hanna et al. 2023), which computes numerical comparisons; the docstring circuit, which routes variable names through attention; and the indirect object pronoun circuit. A consistent finding across circuits is that they involve fewer than 20% of model parameters, suggesting significant redundancy in transformers.

Vision Models

Circuit analysis in ViT is nascent but promising. Conmy et al. 2023 applied ACDC (automated circuit discovery via component pruning) to ViT on image classification tasks. Early results suggest that object-class circuits span layers 6-12 and involve attention heads that attend to foreground patches. The CLS-to-patch attention pattern in layer 11 was identified as a key "readout" mechanism.

Speech Models

In Whisper, the cross-attention from decoder to encoder acts as an alignment mechanism. Research by Gong et al. 2023 identified heads in Whisper-large that function as "phoneme-to-character" circuits: they attend from a character output position to the audio frames containing the corresponding phoneme. These heads are activated only when the model is in "committed" (low-entropy) decoding states, suggesting they are part of a circuit for confident transcription.

Real Model Artifact

Wang et al. 2022 "Interpretability in the Wild: a Circuit for Indirect Object Identification in GPT-2 small" is the most comprehensive circuit analysis published for a real model. Key findings: (1) the circuit involves 26 of GPT-2's 144 attention heads; (2) it achieves 86% of the full model's IOI performance; (3) the circuit generalises across 18 different prompt templates; (4) negative name-mover heads (which suppress the subject name) are a novel head type not predicted by prior theory; (5) the circuit fails on templates where the indirect object position is unusual (e.g., passive voice). The full circuit diagram is available at the paper's interactive website.

Assessment Check

Scenario 1

You run DLA and find that layer 7 has the highest logit attribution for the target token. You then ablate layer 7 and find that target token probability drops only slightly. How do you reconcile these findings?

Strong answer: DLA measures the contribution of a component's output in the actual forward pass, but it doesn't test causality. Ablating layer 7 may have little effect if backup components (e.g., layer 8 name-mover heads) compensate. The high DLA means layer 7 is actively contributing in the normal run, but the circuit is redundant — removing one node doesn't collapse the behaviour. This is exactly why the IOI circuit has "backup" name-mover heads.

Scenario 2

A circuit identified on IOI sentences in English is tested on equivalent sentences in French. Performance drops from 86% to 61%. What are the most plausible explanations?

Strong answer: (1) GPT-2 was trained primarily on English; French token representations are less well-formed, so the name-mover heads may not reliably identify French proper names. (2) The circuit may use surface-level token patterns (e.g., specific byte-pair encodings for English names) rather than abstract structure. (3) Word order differences in French subordinate clauses may shift the positions of S and IO tokens, breaking the positional attention patterns. The key test is to run path patching on French sentences and see if the same 26 heads are still on the critical path.

Superposition

Track 3: Interpretability
5 of 8
MLP Mechanics, Circuits
80 min
Text · Vision · Speech

Learning Outcomes

  • Explain the superposition hypothesis and why it arises in neural networks
  • Distinguish polysemanticity (neuron level) from superposition (geometry level)
  • Describe how Sparse Autoencoders (SAEs) recover monosemantic features
  • Explain the geometry of superposition: how features are packed as nearly-orthogonal vectors
  • Interpret the sparsity-capacity tradeoff and predict when superposition is likely
  • Evaluate SAE reconstruction quality using L0, L1, and explained variance metrics

Conceptual Setup

The superposition hypothesis, introduced by Elhage et al. 2022, proposes that neural networks represent more features than they have dimensions by encoding features as nearly orthogonal directions in activation space rather than as individual neurons. A network with d neurons can represent up to O(d / sparsity) features when those features are sparse — i.e., only a small fraction are active at any given time.

This explains a long-standing puzzle in interpretability: why are individual neurons polysemantic? A neuron in GPT-2 layer 9 might activate for "cat", "dog", "France", and "logarithm" — apparently unrelated concepts. Under superposition, this is not a bug but a consequence of efficient packing: each "true" feature is distributed across multiple neurons, and each neuron participates in multiple features.

Superposition creates a fundamental challenge for interpretability: the neuron basis is not the feature basis. We cannot simply look at what each neuron does. We must find the features — the actual computational units — which live as directions in activation space. Sparse Autoencoders (SAEs) are designed to do exactly this.

Core tension

Models need to represent more concepts than they have dimensions. Superposition is the solution — but it makes interpretation harder because no single neuron = no single concept.

Intuition First

Think of a city's street map. A city has thousands of buildings, but streets run in only a few directions (north-south, east-west, and diagonals). Each building is located using a combination of street coordinates. This is superposition: many buildings (features) encoded using few directions (dimensions). Two nearby buildings share coordinate components — but their full address distinguishes them.

The key requirement for superposition to work is sparsity: at any given time, only a few features are active. If every feature were always on, they would interfere catastrophically. With sparsity, the cross-feature interference averages to near zero.

Sparse Autoencoders are the tool that "de-superpose" activations. An SAE takes a d-dimensional activation vector, expands it to a much larger space (e.g., 4d or 64d), applies a sparsity-inducing activation (ReLU or TopK), then reconstructs the original. The expanded representation isolates individual features into individual dimensions of the wider space.

Technical Core

Superposition geometry: features are vectors f_i ∈ ℝ^d with ||f_i|| ≈ 1. If there are N features and N >> d, they must be nearly orthogonal but not perfectly orthogonal. The interference between features i and j is (f_i · f_j)². With N features in d dimensions, the mean squared interference ≈ N/d². For sparsity p (fraction of features active), the expected loss from interference ≈ p × N/d².

Toy model: d=2, N=5 features
Feature directions (nearly orthogonal in 2D):
  f1 →  [1.0,  0.0]
  f2 →  [0.6,  0.8]
  f3 →  [-0.6, 0.8]
  f4 →  [-1.0, 0.0]
  f5 →  [0.0, -1.0]

With sparsity=0.1, only ~1 feature active at a time:
  interference ≈ 0.1 × 5/4 = 0.125  (manageable)
With sparsity=0.5:
  interference ≈ 0.5 × 5/4 = 0.625  (severe)

Sparse Autoencoder architecture:

Input: x ∈ ℝ^d  (MLP activation at one token position)
   ↓
Encoder: h = ReLU(W_enc · (x - b_pre) + b_enc)  ∈ ℝ^N  (N >> d)
   ↓
Sparse h: most entries zero, few active
   ↓
Decoder: x̂ = W_dec · h + b_pre
   ↓
Loss = ||x - x̂||² + λ||h||₁
         reconstruction    sparsity penalty

SAE training objective: minimise reconstruction loss while encouraging sparsity (L1 penalty on hidden activations). The decoder columns W_dec[:,i] are the "feature directions" — each is a d-dimensional vector in activation space.

MetricMeaningGood Value
L0 (mean active features)Average sparsity of h10–100 per token
L1 norm of hTotal activation magnitudeAs low as possible
Explained varianceFraction of x variance captured by x̂>0.90

Dictionary Learning connection

SAEs are a form of dictionary learning: they find an overcomplete dictionary of features such that any activation can be represented as a sparse combination of dictionary atoms. This is related to ICA (Independent Component Analysis) and sparse coding (Olshausen & Field 1996).

Worked Example

import torch
import torch.nn as nn

# Minimal SAE implementation
class SparseAutoencoder(nn.Module):
    def __init__(self, d_model, d_hidden, l1_coeff=1e-3):
        super().__init__()
        self.W_enc = nn.Linear(d_model, d_hidden, bias=True)
        self.W_dec = nn.Linear(d_hidden, d_model, bias=True)
        self.l1_coeff = l1_coeff
        # Normalise decoder columns to unit norm
        with torch.no_grad():
            self.W_dec.weight.data = nn.functional.normalize(
                self.W_dec.weight.data, dim=0)

    def forward(self, x):
        h = torch.relu(self.W_enc(x))       # sparse hidden
        x_hat = self.W_dec(h)               # reconstruction
        recon_loss = (x - x_hat).pow(2).mean()
        l1_loss    = self.l1_coeff * h.abs().mean()
        return x_hat, h, recon_loss + l1_loss

# Usage: extract MLP layer 9 activations from GPT-2 on a corpus,
# then train the SAE on those activations.
# d_model=768, d_hidden=768*8=6144 is a common choice.
sae = SparseAutoencoder(d_model=768, d_hidden=6144)
print(f"SAE parameters: {sum(p.numel() for p in sae.parameters()):,}")
# 9,447,936 — about 8% of GPT-2-small

Mechanistic Interpretation Layer

Text Models

Anthropic's large-scale SAE analysis of Claude 3 Sonnet (Templeton et al. 2024) found SAE features that are strikingly monosemantic: single features active for "DNA sequences", "the concept of bridge in music", "racist tropes", "the Golden Gate Bridge". One feature activated strongly for thousands of contexts involving the Golden Gate Bridge across languages, modalities, and contexts. Steering the model with this feature caused the model to describe itself as the Golden Gate Bridge — demonstrating causal influence on generation.

Vision Models

Park et al. 2023 applied dictionary learning to ViT activations and found that SAE features in later layers correspond to interpretable visual concepts: "dog ear", "red stripe", "window frame." Early layers yield texture and frequency features. The degree of monosemanticity increases with layer depth, consistent with the text findings.

Speech Models

Superposition in speech models has been studied in wav2vec2 and HuBERT. Research shows that hidden units in these models are polysemantic at the neuron level but that sparse linear decompositions yield features corresponding to phonemic categories, prosodic features, and speaker identity. The sparsity is lower in speech models than text models — audio frames carry dense, continuous information unlike discrete tokens.

Real Model Artifact

Templeton et al. 2024 "Scaling and evaluating sparse autoencoders" (Anthropic) trained SAEs on Claude 3 Sonnet residual stream with expansion factor 8× (d=4096 → N=32768 features). Key findings: (1) 73% of features had a clear human-interpretable label; (2) dead features: <2% after training with auxiliary loss; (3) mean L0 ≈ 25 (only 25 of 32768 features active per token); (4) explained variance >0.95; (5) features include multimodal concepts (same feature activates for the word "cat" and images of cats). The SAE weights and analysis tools are publicly available via Anthropic's interpretability team.

Assessment Check

Scenario 1

You train an SAE on GPT-2 layer 9 MLP activations and find that 40% of features are "dead" (never activate on your evaluation set). What are the most likely causes and how would you fix each?

Strong answer: (1) L1 coefficient too high → features get suppressed during training. Fix: reduce λ. (2) Decoder columns are not unit-normalised → scale degeneracy causes some features to dominate. Fix: add norm constraint. (3) Evaluation set is too small/homogeneous → features that activate on rare inputs appear dead. Fix: increase evaluation set diversity. (4) d_hidden too large for the amount of training data → underdetermined. Fix: reduce d_hidden or train longer.

Scenario 2

An SAE has high explained variance (0.97) but low feature interpretability (only 30% of features have clear labels). Is the SAE useful for interpretability?

Strong answer: High explained variance means the SAE captures the variance faithfully, which is necessary but not sufficient for interpretability. Low label rate could mean: (1) many features are real but encode concepts that humans don't have ready labels for (e.g., syntactic microfeatures); (2) many features are "junk" — artifacts of SAE training that don't correspond to model-used concepts; (3) the labelling method (max-activating examples) is too coarse. The SAE is partially useful: the 30% interpretable features can still be used for steering and circuit analysis. Test usefulness by checking if the interpretable features have causal effects consistent with their proposed labels.
